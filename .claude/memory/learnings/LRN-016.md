## Pattern

Avant de coder un gros module, une session Rodin de challenge de roadmap révèle systématiquement :

1. **Bloquants éditoriaux** invisibles dans un plan technique — photos à sourcer, contenu pédagogique à valider auprès du vrai utilisateur (Marie)
2. **Infrastructure partagée** à extraire en amont — composants (HintButton, GameOverModal) qu'on aurait dupliqués dans N features puis retrofittés dans la douleur
3. **Incohérences de nommage** — "Frise" utilisé pour deux concepts distincts (lecture vs jeu), clarifiés avant de coder
4. **Problèmes de représentation de données** — intervals `range` traités comme des points, `keywords[]` à un seul élément

## Why

Ces problèmes étaient tous présents dans la roadmap initiale mais invisibles depuis une lecture interne. Le regard extérieur adversarial les fait émerger en 30 minutes au lieu de les découvrir en cours de dev après avoir codé 3 phases.

## How to apply

Pour tout module > 3 phases : faire une session `/rodin` de challenge de roadmap avant de commencer le développement. Vérifier particulièrement : (1) contenu éditorial bloquant, (2) composants réutilisés dans N endroits, (3) ambiguïtés de nommage dans les onglets/interfaces.
